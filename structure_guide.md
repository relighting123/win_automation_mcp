# 🦾 Windows Automation MCP 프로젝트 구조 가이드

이 문서는 프로젝트의 전체적인 구조와 각 구성 요소의 역할을 설명합니다. 전문가를 위한 기술적인 설명과 초등학생도 이해할 수 있는 쉬운 비유를 함께 제공합니다.

---

## 1. 전체 아키텍처 (Overall Architecture)

이 프로젝트는 **계층형 구조(Layered Architecture)**를 따르고 있습니다. 사용자의 명령이 실제 컴퓨터 조작으로 이어지기까지 여러 단계를 거칩니다.

1.  **사용자 인터페이스 (Streamlit)**: 사용자가 명령을 입력하는 창구
2.  **지능 계층 (LLM)**: 명령을 해석하고 어떤 도구를 쓸지 결정하는 두뇌
3.  **중계 계층 (MCP Server)**: LLM과 실무 코드를 연결하는 다리
4.  **도구 계층 (Tools & Actions)**: 실제로 클릭하거나 글자를 읽는 손과 눈
5.  **운영체제 계층 (Windows/pywinauto)**: 실제로 명령이 실행되는 바닥

---

## 2. 주요 파일별 역할

### 📁 `LLM/streamlit_app.py`
-   **전문용어**: **프론트엔드 엔트리포인트(Frontend Entry point)** 및 **오케스트레이터(Orchestrator)**. 사용자의 입력을 받고, LLM API와 통신하며, MCP 도구들의 실행 흐름을 제어합니다.
-   **초등학생 설명**: **'리모컨'이자 '통역사'**예요. 여러분이 하고 싶은 말을 적으면, 그걸 컴퓨터가 알아듣는 말로 바꿔서 전달해주는 역할을 해요.

### 📁 `mcp_server.py`
-   **전문용어**: **MCP 호스트 서버(MCP Host Server)**. LLM이 사용할 수 있는 도구들을 정의(Registration)하고, 네트워크 요청을 받아 실제 파이썬 함수를 실행시키는 서버 엔진입니다.
-   **초등학생 설명**: **'장난감 상자 관리자'**예요. 어떤 장난감(도구)들이 있는지 목록을 만들고, 명령이 내려오면 그 장난감을 꺼내서 작동시켜요.

### 📁 `core/app_session.py`
-   **전문용어**: **세션 관리자(Session Manager) 및 싱글톤(Singleton)**. 현재 제어 중인 애플리케이션의 연결 상태(Window Handle)를 유지하고 관리합니다.
-   **초등학생 설명**: **'기억력 좋은 비서'**예요. 지금 우리가 어떤 창을 보고 있는지, 그 창이랑 손을 잘 잡고 있는지(연결) 계속 기억하고 있어요.

### 📁 `tools/ui_vision_tool.py`
-   **전문용어**: **시각 인식 컴포넌트(Vision Recognition Component)**. OCR(광학 문자 인식)이나 이미지 매칭 기술을 사용하여 화면의 특정 위치를 찾아냅니다.
-   **초등학생 설명**: **'돋보기를 든 탐정'**이에요. 화면에서 "로그인 버튼 어디 있지?" 하고 찾아내서 정확한 위치를 알려줘요.

## 3. 주요 함수와 "만약 이 함수가 없다면?"

### 🛠️ `AppSession.connect()`
-   **전문가 버전**: **프로세스 어태칭(Process Attaching)**. 특정 프로세스 ID나 윈도우 타이틀을 기준으로 `pywinauto` 객체를 타겟 창에 바인딩합니다.
    -   **없을 때의 문제**: **고아 프로세스(Orphan Process) 상태**. 제어 대상에 대한 접근 권한(Handle)이 없어 모든 조작 명령이 `SessionError`를 발생시키며 무산됩니다.
-   **초등학생 버전**: **'손잡기'**. 조종하고 싶은 창이랑 손을 꼭 잡는 거예요.
    -   **없을 때의 문제**: 눈앞에 친구가 있는데 **손을 안 잡고 "가자!"라고 하는 것**과 같아요. 아무리 명령해도 친구(창)는 따라오지 않아요.

### 🛠️ `get_mcp_tools()`
-   **전문가 버전**: **도구 인트로스펙션(Tool Introspection)**. 서버로부터 사용 가능한 함수의 스키마(이름, 파라미터 타입) 정보를 가져와 LLM이 호출 가능한 형태로 변환합니다.
    -   **없을 때의 문제**: **컨텍스트 부재(Lack of Context)**. LLM이 자신이 어떤 능력을 가지고 있는지 모르게 되어, 아무런 도구도 호출하지 못하고 일반 대화만 하게 됩니다.
-   **초등학생 버전**: **'가방 속 확인하기'**. 내 가방에 어떤 도구(가위, 풀, 연필)가 있는지 살펴보는 거예요.
    -   **없을 때의 문제**: **눈을 감고 요리를 하려는 것**과 같아요. 주방(컴퓨터)에 도구는 많지만, 내 손에 뭐가 있는지 몰라서 아무것도 못 해요.

### 🛠️ `analyze_app_screen()`
-   **전문가 버전**: **스크린 래스터 분석(Screen Raster Analysis)**. 현재 화면의 스크린샷을 찍어 텍스트 좌표 데이터로 구조화(Data Structuring)합니다.
    -   **없을 때의 문제**: **좌표 불확실성(Coordinate Uncertainty)**. 버튼의 정확한 위치를 모르기 때문에 '묻지마 클릭'을 하거나 아예 클릭을 포기하게 됩니다.
-   **초등학생 버전**: **'지도 그리기'**. 지금 화면이 어떻게 생겼는지 지도를 그리는 거예요.
    -   **없을 때의 문제**: **깜깜한 방에서 보물찾기**를 하는 거예요. 보물(버튼)이 어딘가 있긴 한데, 보이지가 않아서 손만 허우적거리게 돼요.

### 🛠️ `chat_completion` 호출부 (client.chat.completions.create)
-   **전문가 버전**: **추론 엔진 연결(Inference Engine Interaction)**. 사용자 메시지와 현재 상태(Context)를 LLM에게 전달하여 다음 행동(Action)을 결정하게 하는 핵심 루프입니다.
    -   **없을 때의 문제**: **무뇌 상태(Brainless State)**. 시스템의 지능이 사라져, 복잡한 명령을 분석하거나 상황에 맞는 판단을 내릴 수 없습니다.
-   **초등학생 버전**: **'똑똑한 박사님께 물어보기'**. 우리가 명령을 받으면 어떻게 해야 할지 박사님께 물어보는 거예요.
    -   **없을 때의 문제**: **공부 하나도 안 한 인형** 같아요. 시키면 움직여야 하는데, 정작 "어떻게 움직여?"라고 물어볼 박사님이 없어서 가만히 서 있게 돼요.
