# 발표 멘트(초심자 친화) - OpenProvider 기반 Windows 자동화 구조

이 문서는 팀 공유/데모 발표에서 바로 읽을 수 있도록 만든 **발표 스크립트**입니다.  
흐름은 **큰 그림 -> 동작 흐름 -> 세부 구성 -> 운영 포인트** 순서입니다.

---

## 1) 오프닝 (30초)

안녕하세요.  
오늘은 우리 Windows 자동화 시스템이 어떻게 동작하는지,  
개발자가 아니어도 이해할 수 있게 큰 틀부터 설명드리겠습니다.

핵심은 간단합니다.  
사람이 자연어로 요청하면, LLM이 그 요청을 도구 호출로 바꾸고,  
MCP 서버가 실제로 Windows 앱을 조작해 결과를 돌려줍니다.

---

## 2) 큰 그림 설명 (비유 중심, 1~2분)

이 구조를 **“콜센터 + 현장 기사”** 비유로 보면 쉽습니다.

- **사용자/LLM** = 콜센터 상담원  
  - 고객 요청을 이해하고 “무슨 작업을 할지” 결정
- **Tools** = 작업 지시서  
  - “앱 실행”, “저장 버튼 클릭”, “텍스트 입력” 같은 표준 작업 메뉴
- **Actions/Core** = 현장 기사 + 공구함  
  - 실제 창 찾기, 클릭, OCR 인식, 재시도 등 물리 작업 수행
- **Windows 앱** = 실제 작업 대상 현장

즉, 상담원은 직접 공구를 잡지 않고,  
작업 지시서를 통해 기사에게 일을 맡기는 방식입니다.

이렇게 분리해 두면,
1) 고장 원인을 찾기 쉽고,  
2) 새 기능 추가가 쉬우며,  
3) 다른 LLM으로도 프런트만 교체해 재사용할 수 있습니다.

---

## 3) 요청 1건이 처리되는 흐름 (1분)

예를 들어 사용자가 “메모장 열고 안녕하세요 입력해줘”라고 말하면:

1. Streamlit UI가 요청을 받습니다.
2. OpenProvider 기반 LLM이 필요한 Tool을 고릅니다.
3. MCP 서버에 `tools/call` 요청을 보냅니다.
4. 서버 내부 Action이 실제로 앱을 실행하고 텍스트를 입력합니다.
5. 결과를 다시 LLM과 사용자에게 반환합니다.

이걸 택배 비유로 보면:  
**주문 접수 -> 배송 지시 -> 물류센터 처리 -> 배송 완료 알림**과 같습니다.

---

## 4) 세부 구성 설명 (큰 틀 다음에 디테일)

### 4-1. Streamlit + OpenProvider
- 역할: 대화 UI, 모델 호출, Tool 호출 루프 관리
- 핵심 포인트:
  - OpenAI 호환 Base URL/API Key/Model 입력
  - MCP `initialize -> notifications/initialized -> tools/list/call` 순서 준수

### 4-2. MCP 서버 (`mcp_server.py`)
- 역할: 자동화 도구를 외부에 공개하는 게이트웨이
- 핵심 포인트:
  - Tool 등록
  - 설정/로깅/전송방식 관리

### 4-3. Tools 계층
- 역할: LLM이 직접 호출하는 “메뉴판”
- 핵심 포인트:
  - `launch_application`, `click_app_text`, `type_app_text` 등
  - 비즈니스 의미 단위로 제공

### 4-4. Actions 계층
- 역할: 실제 자동화 시나리오 실행
- 핵심 포인트:
  - 포커스 확보, OCR 탐색, 이미지 탐색, 클릭/입력 실행
  - 실패 시 재시도/에러 메시지 일관성 유지

### 4-5. Core 계층
- 역할: 공통 기능(세션, 대기, 재시도, 연결/종료)
- 핵심 포인트:
  - 싱글톤 세션으로 상태 안정화
  - wait/retry 유틸로 sleep 남용 방지

---

## 5) 왜 이 설계가 좋은가 (비개발자 설명용)

이 구조는 식당 운영에 비유할 수 있습니다.

- 손님 응대(LLM)와 조리(Action/Core)를 분리하면  
  주방이 바빠도 응대 체계가 무너지지 않습니다.
- 메뉴(Tools)가 표준화돼 있으면  
  신입 직원도 일관된 방식으로 주문 처리할 수 있습니다.
- 레시피(config/locator/icon registry)가 파일로 분리돼 있어  
  코드 전체를 안 바꿔도 대상 앱에 맞게 조정할 수 있습니다.

---

## 6) 운영 시 꼭 말해줄 리스크/주의점

1. OCR/이미지 기반 클릭은 화면 해상도/테마/배율 영향이 있습니다.
2. 모델은 반드시 Tool Calling을 지원해야 합니다.
3. MCP URL/세션 초기화 순서가 틀리면 도구 목록 조회에서 멈출 수 있습니다.

---

## 7) 마무리 멘트 (20초)

정리하면,  
우리 시스템은 “자연어 요청 -> 도구 호출 -> 안전한 자동화 실행”을  
계층 분리 구조로 구현한 형태입니다.

OpenProvider 기준으로 프런트(모델) 선택의 유연성을 확보했고,  
백엔드 자동화 로직은 그대로 재사용할 수 있게 설계되어 있습니다.

그래서 **확장성, 유지보수성, 운영 안정성**을 동시에 가져가는 구조입니다.
